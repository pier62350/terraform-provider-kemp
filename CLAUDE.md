# terraform-provider-kemp — codebase guide for contributors (human and Claude)

A Terraform provider for the **Progress Kemp LoadMaster** API v2. Built on
`terraform-plugin-framework`, hand-rolled HTTP client, MPL-2.0.

Published at `registry.terraform.io/providers/pier62350/kemp`.

---

## What's where

```
main.go                    Provider entry. Address: registry.terraform.io/pier62350/kemp.
internal/loadmaster/       Hand-rolled JSON-RPC client for Kemp's /accessv2 endpoint.
internal/provider/         Resource + data source implementations (terraform-plugin-framework).
examples/                  Per-resource HCL + import.sh used by tfplugindocs.
docs/                      Generated by tfplugindocs — do not hand-edit.
tools/                     Pins the tfplugindocs binary in its own go.mod.
.goreleaser.yml            Builds binaries for 10 OS/arch combos, signs SHA256SUMS.
.github/workflows/         test.yml on PR/push, release.yml on v* tags.
terraform-registry-manifest.json  protocol_versions=["6.0"], required by the Registry.
```

## Resources at a glance

| Resource | Notes |
|---|---|
| `kemp_virtual_service` | VS + SSL/SNI + ESP + WAF intercept_mode |
| `kemp_sub_virtual_service` | Same surface as VS, created via `modvs createsubvs` |
| `kemp_real_server` | Backend pool member |
| `kemp_certificate` | PEM/PFX upload |
| `kemp_acme_certificate` | Let's Encrypt / DigiCert issuance |
| `kemp_acme_account` | Service-level ACME bootstrap (one-shot) |
| `kemp_match_content_rule` … `kemp_replace_body_rule` | 6 system-level content rules (rule types 0–5) |
| `kemp_virtual_service_rule` | Attach a content rule to a VS by direction |
| `kemp_virtual_service_waf_rule` | Attach a WAF rule (or rule set) to a VS |
| `kemp_owasp_custom_rule` / `…_data` | OWASP/ModSecurity custom files (admin-level) |
| `kemp_waf_custom_rule` / `…_data` | Legacy commercial WAF custom files |

Two data sources: `kemp_virtual_service`, `kemp_real_server`.

**Every resource implements `ImportState`.** Match the pattern of an existing
resource when adding new ones.

## Wire-format quirks — important, do not unlearn

These come from the official spec at https://loadmasterapiv2.docs.progress.com/
and from validation against a live LoadMaster. **The spec is the source of
truth.** Notes on patterns that aren't obvious:

1. **VS reference: bare numeric Index.** `vs: "12"` works for `showvs/modvs/delvs`.
   `vs: "!12"` fails — Kemp parses `!12` as an address and complains `port:
   String Value missing`.

2. **RS reference: bang-prefixed Index.** `rs: "!5"` for showrs/modrs/delrs.
   Without the bang, Kemp tries to look up by address.

3. **`addrs` is the odd one out.** It uses `vs` (bare numeric, *not* bang),
   `rs` (the RS *address*, not Index), and `rsport` (not `port`).

4. **`vsaddwafrule` keys on the VS triplet** (`vs`+`port`+`prot`), not the
   Index. That's why `kemp_virtual_service_waf_rule` takes
   `virtual_service_address/port/protocol` instead of an `id`.

5. **SubVS creation goes through the parent's `modvs`** with
   `createsubvs: ""`. There's no `addsubvs` command. The response includes
   the new SubVS's Index, used for all subsequent CRUD.

6. **PascalCase response fields.** Kemp returns JSON with capitalized keys
   (`Index`, `VSAddress`, `RsIndex`). Use explicit `json:"PascalName"` tags
   on Go structs — don't rely on automatic snake_case conversion.

7. **`status: "fail"` at HTTP 200.** Even on a successful HTTP response,
   Kemp may return `{"code":422,"status":"fail","message":"..."}`. The
   internal `loadmaster.Client.call` already checks for this and surfaces
   `*loadmaster.Error` accordingly.

8. **CertFile is comma-separated** — multiple cert names mean SNI. The
   provider exposes this as a `cert_files` list in HCL and joins on send.

## Where to look up new commands

The full Postman collection is downloadable as JSON; **always consult it
before adding a new resource** — the example bodies show the exact
parameter names Kemp expects. The kreemer/mathlu repos that originally
inspired this provider are now demoted to historical reference only;
their patterns sometimes diverge from current firmware (notably the
`!N` syntax that this codebase deliberately avoids).

```bash
# Re-fetch the spec
curl -sL "https://loadmasterapiv2.docs.progress.com/api/collections/1897577/VUjPK5sb?segregateAuth=true&versionTag=latest" \
  -o ~/kemp-api-spec/collection.json

# Find a command's example
python3 - <<'EOF'
import json, re
with open('/path/to/collection.json') as f:
    c = json.load(f)
def walk(items, path=''):
    for it in items:
        if 'item' in it:
            yield from walk(it['item'], path + '/' + it.get('name',''))
        else:
            yield path + '/' + it.get('name',''), it
for path, req in walk(c['item']):
    body = req.get('request',{}).get('body',{}).get('raw','')
    if '"<CMD_NAME>"' in body:
        print(path); print(body)
EOF
```

Replace `<CMD_NAME>` with `addvs`, `modrs`, `addacmecert`, `vsaddwafrule`,
etc. Each entry includes the full request body, an HTML description with
parameter semantics, and example responses.

## Local dev workflow

### Build and install

```bash
make build                # go build -v ./...
make install              # go install -v ./...
make test                 # go test -v -cover -timeout=120s -parallel=10 ./...
make lint                 # golangci-lint
make generate             # cd tools; go generate ./...   (regenerates docs/)
```

### Use the local build via `dev_overrides`

In `~/.terraformrc`:

```hcl
provider_installation {
  dev_overrides {
    "pier62350/kemp" = "/home/<user>/go/bin"
  }
  direct {}
}
```

Then run `terraform plan/apply` directly — **do not** run `terraform init`
when `dev_overrides` is active (the framework warns about it; init isn't
needed for overridden providers).

### Acceptance tests against a live LoadMaster

```bash
# Required env vars (load from a gitignored ~/.kemp_env or similar)
export KEMP_HOST="https://<host>:9443"
export KEMP_API_KEY="..."          # OR KEMP_USERNAME + KEMP_PASSWORD

make testacc                       # TF_ACC=1 go test -v -timeout 120m ./...
```

Acceptance tests **must not** target addresses that are already in use on
the target LoadMaster. Use a dedicated lab unit and pick fresh VS
addresses (10.x.x.x for examples in this repo, but the actual lab unit's
subnet for live testing).

## Release flow (Terraform Registry)

The provider is signed and published via the GitHub Actions `release`
workflow on every `v*` tag.

### One-time setup (already done for `pier62350/kemp`)

1. **GPG key** — RSA-4096 (Registry rejects ECC), no passphrase. Algorithm,
   key size, and registration with the Registry are all required.
2. **GitHub Secrets** on the repo: `GPG_PRIVATE_KEY` (full armored block
   incl. BEGIN/END) and `PASSPHRASE` (empty if the key has none).
3. **Public key uploaded** to https://registry.terraform.io/settings/gpg-keys
   (paste the output of `gpg --armor --export <fingerprint>`).
4. **Provider connected** to the Registry via the web UI.

### Cutting a new release

```bash
git tag -a vX.Y.Z -m "Release notes summary"
git push origin vX.Y.Z
```

That's it. The `Release` workflow runs goreleaser, signs `SHA256SUMS`,
attaches the artifacts to a GitHub release, and the Registry auto-ingests
within minutes. Tags must follow `vMAJOR.MINOR.PATCH` semver.

### Release artifact inventory (per goreleaser config)

10 platforms × `.zip` + `_SHA256SUMS` + `_SHA256SUMS.sig` + `_manifest.json`
= 13 artifacts uploaded to the GitHub release. Don't add or remove
platforms without checking the Registry's supported list.

## Conventions to follow when contributing

- **Always implement `ImportState`** on new resources. Use the simplest
  sensible ID format: a single name for resources keyed by name, a slash-
  separated tuple for nested ones (`<parent>/<child>` for real_server,
  sub_virtual_service; `<vs>/<direction>/<rule>` for VS↔rule attachment).

- **Go struct field tags**: explicit `json:"PascalName"` on every field.
  The wire format is fixed by Kemp — don't rely on automatic naming.

- **Don't introduce retry/backoff libraries.** The previous attempt
  inherited `cenkalti/backoff/v5` from a reference repo; it was overkill
  for LAN calls. If transient failures become real, add a 10-line stdlib
  retry, not a dependency.

- **Don't depend on third-party Kemp client libraries.** Earlier scaffold
  considered `github.com/kreemer/loadmaster-go-client`, but that repo has
  no LICENSE — using it in a published provider is legally murky. The
  `internal/loadmaster` package is hand-rolled MPL-2.0 for that reason.

- **No emojis, no AI co-author trailers in commits.** Standard hygiene.

- **MarkdownDescription drives the rendered Registry docs.** Keep field
  descriptions accurate, terse, and example-rich; they appear verbatim in
  `docs/<kind>/<name>.md` after `make generate`.
